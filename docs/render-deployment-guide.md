# OutcomeIQ — Render and Vercel Deployment Guide

## Purpose and Scope

This runbook prepares OutcomeIQ for a manual one-month public MVP deployment. It does not contain credentials and does not authorize automatic deployment. The first live version continues to use simulated AI-provider telemetry, so no OpenAI, Anthropic or other real provider key is required.

Before beginning, run `.\scripts\pre_deploy_check.ps1` locally and review the [deployment checklist](deployment-checklist.md).

## 1. Push a Reviewed Revision

1. Confirm the intended branch is tested and the Git working tree is clean.
2. Confirm `backend/.env` and `frontend/.env` are absent from Git status and repository history.
3. Push the reviewed revision to GitHub manually.
4. Record the commit SHA being deployed so the live version can be traced and rolled back.

Repository: [Soham2500/OutcomeIQ](https://github.com/Soham2500/OutcomeIQ)

## 2. Create Managed PostgreSQL on Render

1. In the Render dashboard, create a PostgreSQL database in the region intended for the backend.
2. Use a unique production database name and credentials generated by the provider.
3. Copy the private/internal connection URL into a secure temporary location.
4. Do not paste the URL into this repository, a README, a screenshot or a Vite variable.
5. Configure database backups and retention appropriate to the selected Render plan before accepting non-demo data.

The database must be network-accessible to the Render backend. Prefer Render's internal connection URL when the database and service are in the same region.

## 3. Create the Render Backend Web Service

Create a Web Service from the GitHub repository. Set the service root directory to `backend` so backend-relative commands and files resolve correctly.

### Option A — Existing Dockerfile

- Runtime/deployment method: Docker
- Root directory: `backend`
- Dockerfile: `Dockerfile` relative to that root
- Health-check path: `/api/v1/health`

The committed Dockerfile starts Uvicorn on `0.0.0.0:8000`. Confirm Render detects that listening port during the first deploy. Do not reuse values from the local root `docker-compose.yml`; they are development-only placeholders.

### Option B — Native Python Service

If Docker is not used:

- Root directory: `backend`
- Build command: `pip install -r requirements.txt`
- Start command: `uvicorn app.main:app --host 0.0.0.0 --port $PORT`
- Health-check path: `/api/v1/health`

Use only one deployment method. Docker is closest to the locally verified packaging; native Python can be simpler to inspect during a student MVP deployment.

## 4. Configure Backend Environment Variables

Enter these values in Render's private environment-variable interface:

```text
APP_NAME=OutcomeIQ
APP_ENV=production
DEBUG=false
DATABASE_URL=<Render PostgreSQL URL>
SECRET_KEY=<long random production secret>
ACCESS_TOKEN_EXPIRE_MINUTES=1440
BACKEND_CORS_ORIGINS=<frontend HTTPS URL>
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=<Gmail address>
SMTP_PASSWORD=<Gmail app password>
MAIL_FROM=<Gmail address>
MAIL_FROM_NAME=OutcomeIQ
OTP_EXPIRE_MINUTES=10
OTP_RESEND_COOLDOWN_SECONDS=60
```

Use `backend/.env.production.example` only as a contract. Do not upload local `backend/.env`. Generate a new, long production signing secret and keep it in Render only.

For Gmail SMTP, create an app password in the deployment owner's Google account. Never paste a normal Gmail password into Git, Vercel, frontend variables or screenshots.

If the frontend URL does not exist yet, use a temporary approved HTTPS origin or complete the frontend creation before final validation. Replace it with the exact deployed frontend origin; do not use `*`.

## 5. Deploy and Initialize the Backend

1. Trigger a manual backend deployment for the recorded commit.
2. Review build and startup logs without copying secrets into tickets or documentation.
3. Apply migrations from the backend working directory using the platform's approved shell, job or release-command mechanism:

   ```text
   alembic upgrade head
   ```

4. Seed deterministic demo pricing:

   ```text
   python scripts/seed_pricing_rates.py
   ```

5. Optionally seed synthetic demo data:

   ```text
   python scripts/seed_demo_data.py
   ```

The commands above are conceptual deployment steps; availability of an interactive shell or release job depends on the chosen service plan. Never run migrations implicitly on every application startup. Demo data is optional and must contain no real customer content.

Verify these public URLs:

- `https://YOUR_BACKEND_DOMAIN/api/v1/health`
- `https://YOUR_BACKEND_DOMAIN/docs`

## 6. Deploy the Frontend as a Render Static Site

1. Create a Render Static Site from the same reviewed GitHub revision.
2. Prefer root directory `frontend`.
3. Set `VITE_API_BASE_URL` to `https://YOUR_BACKEND_DOMAIN/api/v1`.
4. Use build command `npm install && npm run build`.
5. If the root directory is `frontend`, set publish directory to `dist`.
6. If the repository root is used instead, use a frontend-aware build command and publish directory `frontend/dist`.
7. Configure a rewrite from `/*` to `/index.html` with status `200` so React Router routes work on direct navigation.
8. Deploy, then copy the final HTTPS frontend origin into backend `BACKEND_CORS_ORIGINS` and redeploy the backend if needed.

## 7. Deploy the Frontend on Vercel

As an alternative to Render Static Site:

1. Import the GitHub repository into Vercel.
2. Set the project root directory to `frontend` and select/detect Vite.
3. Set `VITE_API_BASE_URL=https://YOUR_BACKEND_DOMAIN/api/v1` for the Production environment.
4. Use `npm install && npm run build` as the build command if overriding Vercel's detected commands.
5. Confirm output directory `dist`.
6. Configure an SPA rewrite to `/index.html` for unmatched client-side routes before relying on direct dashboard links.
7. Deploy, then set the exact Vercel HTTPS origin in backend `BACKEND_CORS_ORIGINS` and redeploy the backend.

`VITE_API_BASE_URL` is public browser configuration, not a secret. Never put database URLs, signing keys or provider keys into a `VITE_*` variable.

## 8. Run the Production Smoke Check

From the project root:

```powershell
.\scripts\prod_smoke_check.ps1 -BackendBaseUrl "https://YOUR_BACKEND_DOMAIN" -FrontendBaseUrl "https://YOUR_FRONTEND_DOMAIN"
```

The script checks backend health, Swagger UI and the frontend homepage without registration or login. Continue with the manual authenticated checklist only after it prints `PRODUCTION SMOKE CHECK PASSED`.

## Safety and Rollback Notes

- Do not use local `backend/.env` in production.
- Do not use local Docker Compose credentials in production.
- Use `APP_ENV=production`, `DEBUG=false` and an exact HTTPS CORS origin.
- Change `SECRET_KEY` before deployment and never expose it to the frontend.
- Do not add real AI-provider keys for the first live version.
- Keep the previously working commit SHA and redeploy it if a release fails.
- Treat schema rollback separately from application rollback; do not downgrade a production database without reviewing data-loss risk.
- Run the smoke check again after every backend, frontend, environment or migration change.

Provider interfaces evolve. Confirm field names against the current official [Render documentation](https://render.com/docs) and [Vercel documentation](https://vercel.com/docs) during the manual deployment.
